## CRITICAL: Root Cause of Training Failure

### The completion detection pipeline is fundamentally broken

Training may have **succeeded on the pod** multiple times, but hKask has
**no mechanism to detect it**. Here is the complete failure chain:

#### 1. `map_pod_status` cannot return `Completed`

```rust
// runpod.rs:133-140
fn map_pod_status(status: &str) -> TrainingJobStatus {
    match status {
        "CREATING" | "PENDING" => TrainingJobStatus::Queued,
        "RUNNING" => TrainingJobStatus::Running,
        "FAILED" | "ERROR" | "STOPPED" | "TERMINATED" => TrainingJobStatus::Failed,
        _ => TrainingJobStatus::Queued,
    }
}
```

There is **no mapping to `Completed`**. The function can only return
`Queued`, `Running`, or `Failed`. `TrainingJobStatus::Completed` is
impossible to reach through this function.

**Classification:** IS/Observation/Direct-provenance, confidence 1.0.

#### 2. The pod stays `RUNNING` forever

The install script (generated by `generate_install_script`) ends with:
```bash
exec sleep infinity
```

This keeps the pod alive for SSH debugging. RunPod's `desiredStatus`
remains `RUNNING`. The `status()` function queries `desiredStatus` only,
so it always returns `Running`.

**Classification:** IS/Observation/Direct-provenance, confidence 1.0.

#### 3. `completion_metadata()` always returns `None`

```rust
// runpod.rs:1088-1094
async fn completion_metadata(&self, _job_id: &str) -> Result<Option<CompletionMetadata>, ProviderError> {
    // Runpod doesn't provide structured training metrics via API.
    Ok(None)
}
```

Even if status somehow returned `Completed`, the auto-registration code
in `training_status` (lib.rs:835) calls `self.host.completion_metadata()`,
which always returns `Ok(None)`, hitting the `_ => "No completion metadata
available"` branch.

**Classification:** IS/Observation/Direct-provenance, confidence 1.0.

#### 4. `adapter_weight_path()` always returns `None`

```rust
// runpod.rs:1096-1102
async fn adapter_weight_path(&self, _adapter_id: &str) -> Result<Option<PathBuf>, ProviderError> {
    // Weights are on the Runpod pod — need to be downloaded separately.
    Ok(None)
}
```

The adapter blob is never stored locally because the weight path is
always `None`.

**Classification:** IS/Observation/Direct-provenance, confidence 1.0.

#### 5. The completion manifest is written on the pod but never fetched

The install script writes a completion manifest to a local path on the
pod:
```bash
cat > "$manifest_path" <<MANIFEST
{ "job_id": "...", "status": "success", ... }
MANIFEST
```

But:
- The manifest path comes from `artifacts.completion_manifest_path`,
  which is set to `jobs/{job_id}/completion-manifest.json` (a HuggingFace
  repo path, not a local filesystem path).
- The install script treats this as a local path. With CWD at `/` (Docker
  default), `cat > "jobs/{job_id}/completion-manifest.json"` fails because
  `/jobs/` doesn't exist. With `set -euo pipefail`, this **exits the
  script with an error** — potentially after training already succeeded.
- Even if the manifest were written, nobody reads it. The
  `fetch_completion_manifest` method exists in `huggingface.rs` but is
  **never called** from the status or completion detection path.
- The install script uploads the adapter to HuggingFace but does NOT
  upload the manifest.

**Classification:** IS/Observation/Direct-provenance, confidence 0.95.

#### 6. The auto-registration code in `training_status` is dead code

```rust
// lib.rs:821
if status == TrainingJobStatus::Completed {
    // ... auto-register adapter ...
}
```

Since `map_pod_status` can never return `Completed`, this entire block
(80+ lines of adapter registration, A/B comparison, blob storage) is
**dead code**. It has never executed in production.

**Classification:** IS/Observation/Direct-provenance, confidence 1.0.

### Summary: Why Training "Has Never Worked"

The training pipeline may have actually **succeeded** on RunPod pods —
the install script runs `axolotl train`, which may have completed, and
the adapter may have been uploaded to HuggingFace. But hKask **cannot
detect completion** because:

1. Pod status is always `RUNNING` (kept alive for debugging)
2. `map_pod_status` has no `Completed` mapping
3. `completion_metadata()` always returns `None`
4. The completion manifest is never fetched from the pod or HuggingFace
5. The manifest write may fail (path mismatch), causing the script to
   exit with an error AFTER training succeeded

The $600 billing leak was a symptom: pods kept billing because they were
never detected as complete, and the in-memory pod tracking lost the IDs
on restart. The post-mortem fixed the pod ID persistence but did NOT fix
the completion detection.

### Required Fix (Minimum Viable)

The status function must check for completion by **fetching the
completion manifest from HuggingFace** (since the adapter is already
uploaded there). The flow should be:

1. `status()` queries RunPod for pod `desiredStatus` (still `RUNNING`)
2. If pod is `RUNNING`, also try to fetch the completion manifest from
   HuggingFace via `fetch_completion_manifest()`
3. If manifest exists and `status == "success"`, return `Completed`
4. If manifest exists and `status == "failed"`, return `Failed`
5. If manifest doesn't exist, return `Running` (still in progress)

Additionally, the install script must:
1. Write the manifest to `/workspace/completion.json` (local, guaranteed
   to work)
2. Upload the manifest to HuggingFace at `jobs/{job_id}/completion-manifest.json`
3. Only then `exec sleep infinity`

**Classification:** OUGHT/Prohibition (this is a correctness bug, not a
  feature gap). Provenance = direct (codebase audit). Confidence 1.0.

## Open Questions — Resolved

### Q1: Does RunPod `/workspace` survive pod restart?

**RESOLVED: YES.**

RunPod documentation (docs.runpod.io/pods/storage/types) states:
> "Data stored in the /workspace directory survives Pod stops and
> restarts, but is deleted when the Pod is terminated."

Network volumes persist independently and can be reassigned to different
pods. The key distinction is **stop** (volume persists) vs **terminate**
(volume deleted). RunPod's `podFindAndDeployOnDemand` creates on-demand
pods; container restarts preserve `/workspace`.

**Impact on plan:** Checkpoint resume is feasible. Checkpoints saved to
`/workspace/outputs/{job_id}/checkpoint-{step}/` survive pod restarts.
The `resume_from_checkpoint` parameter can point to this path.

**Classification:** IS/Observation/External-provenance, confidence 0.95.
  Source: RunPod official docs.

### Q3: Axolotl checkpoint format

**RESOLVED: safetensors, compatible with hKask.**

Axolotl saves checkpoints as `adapter_model.safetensors` in
`checkpoint-{step}/` directories under `output_dir`. The
`resume_from_checkpoint` parameter is supported natively. hKask's
completion marker is already `adapter_model.safetensors`
(harness.rs:219).

Axolotl also supports `auto_resume_from_checkpoints: true` in the YAML
config, which automatically resumes from the latest checkpoint if one
exists.

**Impact on plan:** No format conversion needed. The checkpoint format
is already compatible. The gap is that hKask doesn't set
`resume_from_checkpoint` or `auto_resume_from_checkpoints` in the
rendered config.

**Classification:** IS/Observation/External-provenance, confidence 0.90.
  Source: Axolotl docs (docs.axolotl.ai/docs/config-reference.html),
  GitHub issue #2149.

### Q9: Does TRL's GRPOTrainer work without vLLM?

**RESOLVED: NO, vLLM is required.**

TRL's GRPOTrainer requires vLLM for generation during RL training. Two
modes are supported:
- **Server mode**: vLLM runs as a separate process on separate GPUs
- **Colocate mode** (default): vLLM runs inside the trainer process,
  sharing GPU memory with the training model

On a single GPU, colocate mode is the only option, but it "may lead to
memory contention on the training GPUs" (HuggingFace docs). For large
models (27B+), this may not fit on a single H100.

hKask's matrix says TRL GRPO is "deferred — requires vLLM". This is
**correct** — TRL GRPO does require vLLM, and the colocate mode may not
work for large models on a single GPU.

**Impact on plan:** The matrix entry for TRL GRPO is accurate. No
update needed for this specific entry. However, Axolotl now also
supports GRPO (via `rl: grpo`), so the Axolotl GRPO entry should change
from ❌ to ✅.

**Classification:** IS/Observation/External-provenance, confidence 0.90.
  Source: huggingface.co/docs/trl/grpo_trainer,
  huggingface.co/docs/trl/vllm_integration, huggingface.co/blog/vllm-colocate.

### Q10: Is the Unsloth `messages` vs `text` format mismatch still blocking?

**RESOLVED: Probably not blocking for Axolotl; was an Unsloth-specific issue.**

The dataset normalization pipeline (`dataset.rs`) normalizes all input
formats (ChatML, ShareGPT, Alpaca, raw text, preference formats) to
canonical `ChatConversation { messages: Vec<ChatMessage> }` where each
`ChatMessage` has `role` and `content`. The normalized output is JSONL
with `{"messages": [{"role": "...", "content": "..."}]}`.

The Axolotl template uses `type: chat_template`, which expects exactly
this `messages` format. So the dataset format is **compatible with
Axolotl**.

The readiness doc (2026-07-10) said "The rendered harness expects `text`;
normalized records contain `messages`" — this was specific to the
**Unsloth** harness, which expected a `text` field. Unsloth is not one
of the three supported harnesses (Axolotl, TRL, Ludwig) in the current
code. The mismatch may have been from an earlier iteration that used
Unsloth directly.

**Impact on plan:** Not a current blocker for Axolotl/TRL/Ludwig. If
Unsloth is added as a 4th harness, the dataset pipeline would need a
`text` format option (format `messages` through the model's chat
template).

**Classification:** IS/Observation/Direct-provenance, confidence 0.85.

### Q15: Is the $600 billing leak truly fixed?

**RESOLVED: The pod ID persistence is fixed, but the ROOT CAUSE is not.**

The post-mortem fixed pod ID persistence (SQLite + JSON file). Pod IDs
now survive process restarts. `drain_all_pods()` can terminate all known
pods on shutdown.

However, the **root cause of the billing leak** was not just pod ID
persistence — it was that **pods were never detected as complete** and
thus never cancelled. Even with perfect pod ID persistence, if
`training_status` never returns `Completed`, the pod keeps billing
forever. The persistence fix lets you cancel pods you know about, but
you still need to manually decide to cancel them because the system
can't detect completion.

The **true fix** requires the completion detection pipeline described
in the "Required Fix" section above.

**Classification:** IS/Observation/Direct-provenance, confidence 0.95.

## What to Do RIGHT NOW (before any plan execution)

### Step 1: Fix the completion detection bug (P0)

This is the single most important fix. Without it, no training
investment matters — you'll never know if training succeeded.

1. In `generate_install_script` (runpod.rs): write manifest to
   `/workspace/completion.json`, then upload it to HuggingFace:
   ```bash
   huggingface-cli upload "$MODEL_REPO" /workspace/completion.json \
       --revision main --path-in-repo "jobs/$JOB_ID/completion-manifest.json"
   ```
2. In `RunpodHost::status()`: if pod is `RUNNING`, attempt to fetch
   the completion manifest from HuggingFace. If found, return
   `Completed` (if status=success) or `Failed` (if status=failed).
3. In `RunpodHost::completion_metadata()`: parse the fetched manifest
   and return `CompletionMetadata` with loss, training_duration, etc.
4. In `RunpodHost::adapter_weight_path()`: return the HuggingFace
   adapter path (the adapter is already uploaded by the install script).

### Step 2: Fix the template wiring gap (P1)

In `AxolotlHarness::render_config()` (harness.rs), add to the context
map:
```rust
("bf16", json!(p.advanced.bf16)),
("gradient_checkpointing", json!(p.advanced.gradient_checkpointing)),
("flash_attention", json!(matches!(p.advanced.attn_implementation.as_deref(), Some("flash_attention_2"))),
("sample_packing", json!(p.sequence.sample_packing)),
```

Then update the template to use these instead of hardcoded values.

### Step 3: Fix the manifest path (P1)

Change `generate_install_script` to always write to
`/workspace/completion.json` (local), then upload to HuggingFace at
the repo path. Don't use the HuggingFace repo path as a local
filesystem path.
