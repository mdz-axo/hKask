## 3. What Remains — Phase 5: CNS Integration

### HIGH — `WalletBackedBudget` in `hkask-cns`

**What:** Create a new `EnergyBudget` variant that is backed by a wallet's rJoule balance instead of periodic replenishment cycles. When gas is consumed, it converts to rJoules via `WalletManager::gas_to_rjoules()` and debits the wallet.

**Where:** 
- **NEW:** `crates/hkask-cns/src/wallet_budget.rs` — `WalletBackedBudget` struct
- **MODIFIED:** `crates/hkask-cns/src/energy.rs` — add `WalletBacked` variant or integrate as separate type
- **MODIFIED:** `crates/hkask-cns/src/energy_budget_management.rs` — add `WalletEnergyEstimator`

**Dependencies:** `hkask-wallet` must be added as a dependency of `hkask-cns` in `crates/hkask-cns/Cargo.toml`. This is safe — CNS depends on wallet, not the reverse.

**Strategy:**
```rust
pub struct WalletBackedBudget {
    pub wallet_id: WalletId,
    pub key_id: Option<ApiKeyId>,
    pub spending_limit_rj: Option<RJoule>,
    pub wallet_manager: Arc<WalletManager>,
    pub gas_per_rjoule: u64,
    pub hard_limit: bool,  // always true for wallet budgets
}
```

The `WalletBackedBudget` should:
- Call `wallet_manager.can_afford(wallet_id, cost_rj)` before operations
- Call `wallet_manager.reserve_rjoules()` before tool execution
- Call `wallet_manager.settle_rjoules()` after tool execution (actual cost may differ from reserved)
- Reject operations when balance insufficient or key spending limit exceeded

**Integration point:** `GovernedTool::invoke()` in `crates/hkask-cns/src/governed_tool.rs` already checks `EnergyBudget` before tool execution. The `WalletBackedBudget` should implement the same check interface so it slots into the existing membrane.

### HIGH — `WalletEnergyEstimator`

**What:** Wrap the existing `CompositeEnergyEstimator` and add rJoule conversion. The estimator produces gas costs; the wallet budget converts to rJoules at reserve/settle time.

**Where:** `crates/hkask-cns/src/energy_budget_management.rs`

**Strategy:**
```rust
pub struct WalletEnergyEstimator {
    inner: CompositeEnergyEstimator,
    gas_per_rjoule: u64,
}

impl EnergyEstimator for WalletEnergyEstimator {
    fn estimate_cost(&self, server: &str, tool: &str, params: &ToolParams) -> EnergyCost {
        self.inner.estimate_cost(server, tool, params)
        // Conversion to rJoules happens at reserve/settle time in WalletBackedBudget
    }
}
```

**Where:** Span emission should happen in the wallet crate (`hkask-wallet`) at each operation site. The wallet crate already depends on `hkask-types` which provides `NuEventSink`. Add span emission calls to:

## 5. Key Decisions to Preserve

1. **Dependency direction: CNS → wallet, NOT wallet → CNS.** `hkask-cns` depends on `hkask-wallet`, not the reverse. The wallet crate does NOT depend on `hkask-cns`. CNS span emission in the wallet crate goes through `hkask-types::event::NuEventSink`, not through `hkask-cns` directly. This prevents circular dependencies.

2. **`WalletBackedBudget` is a variant of `EnergyBudget`, not a replacement.** The existing gas system (periodic replenishment, dimensionless gas units) continues to work for non-wallet-backed agents. `WalletBackedBudget` is an additional budget type that converts gas to rJoules and debits a wallet. Both coexist in the `GovernedTool` membrane.

3. **Per-operation key loading in `signing.rs`.** Treasury keys are derived via HKDF at the moment of signing, used, and zeroized on drop. No long-lived treasury key material exists in memory. This follows the Turnkey/1Password security pattern. Do not change this to long-lived keys without explicit security review.

4. **`LoadedKey` with redacted Debug (MUST-2).** The `LoadedKey` struct in `signing.rs` wraps `Zeroizing<[u8; 32]>` and has a custom `Debug` impl that shows `[REDACTED]`. This prevents key material from appearing in logs, error messages, or debug output. Do not add `Debug` derives that would expose key bytes.

5. **API keys are Ed25519 keypairs, not opaque bearer strings.** The private key IS the API key — returned to the user once at creation time, never stored by hKask. Only the public key is persisted in the `api_keys` table. Authentication works by deriving the public key from the presented private key and looking it up. Do not change this to random hex strings or JWTs.

6. **rJoule is an internal accounting unit, not an on-chain token.** rJoule exists only in hKask's SQLite database. It is pegged to USDC at a configurable rate (`rj_per_usdc`). Do not create an ERC-20, SPL, or HTS token for rJoule.

7. **Hinkal Access Tokens are accepted pre-minted, not minted by hKask.** zkSNARK proof generation (Groth16 circuits, Circom, trusted setup) belongs in the Hinkal SDK, not in a Rust agent container. hKask accepts pre-minted Access Tokens as configuration.

8. **The hKask wallet is a specialized sub-wallet.** It only does what hKask needs: receive deposits, track rJoule balances, issue API keys, process withdrawals. The user's primary wallet (Phantom, HashPack, MetaMask) handles key storage, multi-chain asset management, and DeFi interactions. Do not expand the wallet's scope to general-purpose crypto wallet features.

9. **Chain port implementations are feature-gated.** `solana.rs`, `hedera.rs`, `hinkal.rs` are behind Cargo features (`solana`, `hedera`, `hinkal`). The default build has zero chain SDK dependencies. This keeps the compile-time attack surface minimal.

10. **Forbidden dependencies:** No `openssl` (use `rustls`), no `libsodium` (use Rust-native crypto), no `ring` (use `ed25519-dalek` + `sha2` + `hmac`), no proc-macro crates beyond `thiserror`, `serde`, `async-trait`.
