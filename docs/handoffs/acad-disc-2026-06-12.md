# Handoff — Academic Corpus Discovery Pipeline

**Date:** 2026-06-12
**Session:** Built end-to-end academic author corpus discovery (Semantic Scholar, arXiv, web, YouTube transcripts) with CLI-driven execution, interactive curation, and OCR fallback.

---

## 1. Session Context

Built the full pipeline for discovering an academic author's body of work and generating a style corpus. Started from the `replica_explain` tool's "Planned" academic_author status. Implemented research providers, a `DiscoveryService` orchestrating via MCP dispatch, CLI command with interactive curation, PDF→text extraction with OCR fallback, and documentation updates. The pipeline is functional end-to-end: `kask style discover "David Dunning"` → `kask style embed-corpus`. ~85% complete — concept extraction, method inference, and corpus augmentation remain.

## 3. What Remains

### HIGH — Corpus Augmentation
- **What:** When `kask style discover` finds an existing `{author}/corpus.yaml`, prompt user: "Found existing corpus. [A]ugment or create [N]ew?" If augment, load existing YAML, merge new works (dedup by URL), regenerate. Existing cache preserved.
- **Where:** `crates/hkask-cli/src/commands/discover.rs` — add interactive prompt before calling `DiscoveryService::discover()`. `crates/hkask-services/src/discover.rs` — `augment` field already declared on `DiscoverRequest`, logic not implemented.
- **Strategy:** Check if `{author_slug}/corpus.yaml` exists before discovery. If yes and not `--no-curate`, prompt. If augment, pass `augment: true` to `DiscoverRequest`. `DiscoveryService` loads existing config, merges.

### MEDIUM — Concept Extraction + Method Inference (LLM)
- **What:** After content extraction, run LLM-based concept extraction (semantic deduplication of key ideas from paper titles/abstracts) and method inference (stylometric patterns from sample passages). Feed results into corpus YAML.
- **Where:** Templates exist at `registry/templates/replica/extract-concepts.j2` and `infer-methods.j2`. Need CLI integration — either as additional phases in `DiscoveryService` or as separate post-discovery commands.
- **Dependencies:** Requires inference port (available via `AgentService`). Templates use `reasoning_local` model tier.
- **Strategy:** Add optional phases to `DiscoveryService::discover()` gated by `include_methods` flag (default true per user's instruction: "quality & precision should be default, being cheap should be an option").

### MEDIUM — Registry Management Tool (`kask list` / `kask rm`)
- **What:** General artifact management. `kask list styles` lists all built corpora. `kask rm styles/hemingway` removes a corpus (embeddings + triples + cache + YAML). User specified format: `kask list [registry name]` and `kask rm [registry name]-[artifact name]`.
- **Where:** New CLI command in `crates/hkask-cli/src/commands/`. Could reuse `replica_registry` MCP tool logic or call `EmbeddingStore` directly.
- **Strategy:** `kask list styles` queries `EmbeddingStore` for `style:*:centroid` entries. `kask rm styles/<name>` purges by prefix `style:<name>:` (embeddings + triples), removes cache dir and YAML.

### LOW — Non-ASCII slug handling
- **What:** `slugify()` produces empty strings for all-non-ASCII titles. Edge case for academic authors (titles are overwhelmingly ASCII).
- **Where:** `crates/hkask-services/src/discover.rs` — `slugify()` function.
- **Strategy:** Fallback to UUID-based slug if `slugify()` returns empty.

## 5. Key Decisions to Preserve

1. **CLI is canonical execution path, manifest is specification.** The `replica-discovery.yaml` FlowDef documents the intended pipeline. Actual execution is `kask style discover` → `DiscoveryService` → MCP dispatch. This mirrors the literary pipeline (`kask style embed-corpus` → `EmbedService`). Do not build a manifest execution trigger without understanding why CLI→service was chosen.

2. **DiscoveryService uses MCP dispatch, not direct HTTP.** After finding that `web_search.rs` already starts the MCP server and calls its `web_search` tool (which fuses ALL providers with RRF), the duplicate HTTP functions (`search_semantic_scholar`, `search_arxiv`, `search_web_serpapi`) were deleted. YouTube transcript search is the exception — it requires a two-step SerpAPI flow (search → transcript) that the MCP tool doesn't replicate.

3. **Web search terms are derived from academic titles or user-provided.** The `extract_search_terms()` function does stopword-filtered frequency analysis on paper titles. User can override with `--search-terms`. The web/YouTube search functions take a `query` parameter — not hardcoded to "faculty page institutional profile interview".

4. **Curation is default, agentic is opt-out.** Per Magna Carta P4 (Affirmative Consent), web and YouTube results are presented for user confirmation before inclusion. `--no-curate` skips this. Academic sources (Semantic Scholar, arXiv) are auto-included.

5. **OCR fallback uses local Ollama with LightOnOCR-2:1b.** Default model `maternion/LightOnOCR-2:1b`, overridable via `HKASK_OCR_MODEL`. If model not found, error includes download link: `https://ollama.com/maternion/LightOnOCR-2:1b`. OCR parameters are fully deterministic (temperature=0.1, top_p=1.0, top_k=1, all penalties=0.0).

6. **`kask list` / `kask rm` design:** `kask list [registry name]` lists artifacts. `kask rm [registry name]-[artifact name]` removes. Format uses hyphen separator between registry name and artifact name. This is a general tool, not specific to styles — should work for any registry (styles, bots, templates, etc.).
