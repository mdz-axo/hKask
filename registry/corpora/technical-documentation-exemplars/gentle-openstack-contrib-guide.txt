## First Timers

One of the best ways to start contributing to OpenStack documentation is to walk through the Installation Guide and complete it by hand. Keep notes as you go, and offer suggestions for improvement by filing documentation bugs at Launchpad.

### Setting up for contribution

To get started, first set up your OpenStack account and then set up Gerrit.

**Set up a text editor.** Use any text editor of your choice to work with the documentation. It is a good idea to choose an editor that supports spell checking and syntax highlighting for RST. To keep the documents clean and easy to compare, all of the OpenStack projects require that text is wrapped at 79 characters maximum, with no white spaces at the end of the line. You can configure the text editor to do that automatically.

**Set up git and git-review.** Install git. If you use Windows to contribute to OpenStack, install Git for Windows. In the subsequent procedures, run commands from the Git Bash console. Install git-review so that you are able to submit patches.

**Set up SSH.** On the computer which you commit from, generate an SSH key with `ssh-keygen -t rsa`. Optionally, enter a password. If you enter one, remember it because you must enter it every time you commit. View and copy your SSH key, then sign into gerrit at https://review.opendev.org/. In the upper right, click your username. Click the Settings > SSH Public Keys page. Click Add Key. Paste the key in the Add SSH Public Key web form and click Add.

**Set up a repository.** For the instructions on how to set up a repository so that you can work on it locally, refer to the Starting Work on a New Project of the Infrastructure manual.

### Finding a bug

After setting up your environment for contribution, it is time to find something to work on:

1. Go to the bug lists at Bugs:openstack-manuals for documentation bugs related to the openstack-manuals repository.
2. For project-specific documentation bugs, go to https://bugs.launchpad.net/.
3. Find a bug that has been confirmed by another contributor and assign it to yourself.

### Committing a change

1. Update the repository and create a new topic branch as described in the Starting a Change section of the Infrastructure manual.
2. Fix the bug in the docs.
3. Read Writing style, also pay attention to RST conventions.
4. Create your commit message.
5. Create a patch for review.opendev.org following the Submitting a Change for Review instructions.
6. Follow the URL returned from git-review to check your commit at https://review.opendev.org/<COMMIT-NUMBER>.
7. In Gerrit, wait for the automatic Zuul checks to succeed.
8. Celebrate and wait for reviews!

### Responding to requests

After you submit a patch, reviewers may ask you to make changes before they approve the patch. To submit changes to your patch:

1. Copy the commit number from the review.opendev.org URL.
2. At the command line, change into your local copy of the repository.
3. Check out the patch: `git review -d <COMMIT-NUMBER>`
4. Make your edits.
5. Commit the changes using the amend flag: `git commit -a --amend`
6. Ensure that the Change-ID line remains intact in your commit message. This prevents Gerrit from creating a new patch.
7. Push the changes to review as described in the Updating a Change section of the Infrastructure manual.
8. Wait for more reviews.

### Troubleshooting your setup

**git and git review**

- *Authenticity error.* The first time that you run git review, you might see: "The authenticity of host '[review.opendev.org]:29418' can't be established." Type yes (all three letters) at the prompt.
- *Gerrit connection error.* When you connect to gerrit for the first time, you might see: "Could not connect to gerrit. Enter your gerrit username:" Enter the user name that matches the user name in the Settings page at review.opendev.org.
- *Not a git repository error.* If you see: "fatal: Not a git repository (or any of the parent directories): .git" — change into your local copy of the repository and re-run the command.
- *Gerrit location unknown error.* If you see: "We don't know where your gerrit is. Please manually create a remote named 'gerrit' and try again." — You need to make a git remote that maps to the review.opendev.org SSH port for your repository.
- *Remote rejected error.* If you see: "missing Change-Id in commit message footer" — The first time you set up a gerrit remote and try to create a patch, you may see this message because the tool needs one more edit of your commit message in order to automatically insert the Change-Id. When this happens, run `git commit -a --amend`, save the commit message and run `git review -v` again.
- *Permission denied error.* If you see: "Permission denied (publickey)." — Double check the Settings page at review.opendev.org to make sure your public key has been copied to SSH Public Keys. If you still receive a Permission denied message, you might need to check if you hit "no mutual signature algorithm". Any system running OpenSSH 8.8 or newer disables RSA signatures using the SHA-1 hash algorithm by default. Try using ed25519 to generate your key and upload id_ed25519.pub.
- *Merge conflict, rebase required.* Sometimes your change creates a merge conflict with another patch. In this case, you need to rebase your commit on top of the current state of the repository. In many cases, this can be achieved by pressing the Rebase button in the gerrit interface. If this fails, you must rebase manually.
- *FAILURE in the Zuul check section.* Click the link next to the FAILURE test. Verify the output of the console.html: If it contains errors or warnings related to your commit with a problematic file name and line in question, fix the issues and push the new changes on review. If the issue is not related to your commit, re-run the gate jobs by leaving the word recheck in the comments section of your Gerrit review.

**Network.** If your network connection is weak, you might see: "Read from socket failed: Connection reset by peer" — Try again when your network connection improves. If you suspect that SSH over non-standards ports might be blocked, you can configure git-review to use an https endpoint instead of ssh.

**Python.** If you see: "/usr/bin/env: python: No such file or directory" — Your Python environment is not set up correctly. See the Python documentation for your operating system.

**i18n.** If you see errors with `git review -s` related to commit-msg hooks, you may have a LANGUAGE variable setup to something else than C. Try using instead: `LANG=C LANGUAGE=C git review -s`
